VSS

队友做的，自己比赛时没做出来
自己忽略了一个问题，就是sub_4011B1的栈里保存了完整的（小于1024byte）的输入，而sub_40108E是存在漏洞的，但是只能刚好覆盖ret地址，因此要利用sub_4011B1的栈。

sub_4011B1 . 8B Trash*72B Trash2*8B RopChain*944B
sub_40108E Trash*72B ret

抬高rsp之后，再ret就能直接ret到我们的ropchain
布置栈’A’*72 + (add rsp 0x58; ret;) + ‘A’*8 + ropchain

用ROPGadget这个软件可以自动生成ropchain。真是牛逼。。
分析这次它生成的ropchain，把.data段地址放入rsi，/bin/sh放入rax，把/bin/sh写入.data前8个字节；同理把0x0000000000000000写入.data+8；然后把三个参数rdi,rsi,rdx分别设为data,data+8,data+8（即参数filename,{NULL},{NULL}）,rax设为59(即execve的syscall号)，然后调用syscall

FB

这是一个Off-By-One的堆溢出
signed int64 fastcall sub_40085D(char *str, unsigned int len)
{
char buf; // [sp+17h] [bp-9h]@2
unsigned int i; // [sp+18h] [bp-8h]@1
int v5; // [sp+1Ch] [bp-4h]@1

  v5 = 0;
  for ( i = 0; i <= len; ++i )
  {
    if ( read(0, &buf, 1uLL) < 0 )
      return 0xFFFFFFFFLL;
    if ( buf == 10 )
    {
      str[i] = 0;
      return i;
    }
    str[i] = buf;
  }
  str[i - 1] = 0;
  return i - 1;
}

其中<=len是一个offbyone漏洞。
将下一个chunk的prev_inuse标志位覆盖为0，让下一个chunk认为它之前的这个chunk已经被释放，当它被释放的时候，就会向前合并(backward coalesce)（有人翻译成向后）
合并，则会引发前一个chunk的unlink。

我们无法修改其chunk head，要怎么利用其unlink？用到了linux内存管理的另一个性质，即在free chunk的末尾有chunk size副本，用于便于合并。
修改此size副本变小一点，就可以欺骗next chunk，让我们可以在原来的user data域伪造chunk head。此处我们将size减小为0xf0，减小了8字节，正好跳过了原先chunk里不可控的8字节。

unlink容易出现不可访问地址的问题，直接利用unlink修改GOT表不容易成功，于是迂回利用别的修改内存的函数。unlink负责让list[0]->addr指向&list[0]-18，接下来我们便可以利用内存修改的函数直接修改&list[0]-18起的一大段内存了

这样写的话，保证了只会修改*ptr，并把*ptr改成ptr-0x18
payload += p64(ptr-0x18) + p64(ptr-0x10)
于是我们的list[0].addr便指向了&list[0]-0x18的位置，

接下来我们便可以自由地将list[0].addr写成got表，然后修改got表了。

将free_got改为printf，第七个块内容改成%17$p，然后free第七个块，free已经变成printf，则会把%17$p字符串当做格式化字符串
%17$p，rdi为格式化字符串，再去掉rsi,rax,rbx,r8,r9五个寄存器，那么输出的是从%rsp算起的第12个QWORD，即[rsp + 11]，偏移了88个字节，当前函数rsp减少了0x10，去掉栈帧、ret地址，main函数rsp减少了0x30，去掉栈帧，加起来刚好88字节。因此%17$p刚好能输出返回到__libc_start_main的地址

这样我们能计算出__libc_start_main和system的地址。

([:-1]是因为该函数会自动补一个\x00)

接下来我们可以再将free_got重新改成system，随便一块（比如第6块）改成/bin/sh;，free第6块，便会执行system(“/bin/sh;”)

关于malloc的大小，涉及到unlink大小，第一块选择使用248byte，因为刚好使用了256byte的chunk，8byte的head，这样刚好能覆盖到下一chunk，如果是248byte的话就覆盖不到了。
第二块选择使用240byte，大一点也可以。但如果232byte的话，则会选择更小的chunk了。而且小于0x100字节，由于我们覆盖的LSB，如果size大于0x100字节（比如0x101字节），覆盖掉标志位之后刚好为0x100，但如果小于0x100了，覆盖LSB之后size就变成了0，所以这里不能比240更小了。
